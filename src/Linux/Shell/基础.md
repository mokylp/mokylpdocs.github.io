# Shell语法基础 
## 摘要与学习目标
本报告将系统地讲解Shell脚本编程的核心语法元素，帮助已掌握基本命令的学习者深入理解Shell的语言本质[[[1](https://blog.csdn.net/m0_74907475/article/details/145231855)][[2](https://wxler.github.io/2021/02/10/111447/)][[3](https://blog.csdn.net/zhengnianli/article/details/131200939)][[4](http://www.runoob.com/linux/linux-shell.html)]][[1](https://blog.csdn.net/m0_74907475/article/details/145231855)][[2](https://wxler.github.io/2021/02/10/111447/)][[3](https://blog.csdn.net/zhengnianli/article/details/131200939)][[4](http://www.runoob.com/linux/linux-shell.html)]。通过本教程，您将掌握变量系统、条件判断、循环结构、函数定义、数组操作以及I/O重定向等核心语法特性。每个知识点都配备逐步完善的生产级代码示例，注释详尽，可直接用于实际项目。学习过程中将特别强调常见的避坑点和最佳实践，帮助您避免隐藏的bug和性能问题。

## 第一章 Shell脚本的本质与执行机制
### 理解Shell脚本的基本结构
Shell脚本本质上是一个文本文件，其中包含一系列shell命令[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[5](https://blog.csdn.net/m0_73345433/article/details/136459603)]]。当您执行脚本时，Shell解释器会逐行读取这些命令并执行[[[2](https://wxler.github.io/2021/02/10/111447/)]]。理解这个基本概念对于正确编写和调试脚本至关重要。

每个Shell脚本都以一行特殊的注释开始，称为"Shebang"或"magic number"[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[5](https://blog.csdn.net/m0_73345433/article/details/136459603)]]。这一行指定了用来执行脚本的解释器[[[4](http://www.runoob.com/linux/linux-shell.html)]]。虽然这一行以`#`开头看起来像注释，但它具有特殊的含义[[[2](https://wxler.github.io/2021/02/10/111447/)]]。当您使用`./script.sh`方式直接执行脚本时，操作系统会读取第一行，根据其中指定的路径来选择合适的解释器[[[2](https://wxler.github.io/2021/02/10/111447/)]][[[4](http://www.runoob.com/linux/linux-shell.html)]]。这意味着如果您通过`bash script.sh`或`sh script.sh`方式执行脚本，系统会忽略第一行，直接使用指定的解释器[[[2](https://wxler.github.io/2021/02/10/111447/)]]。

```bash
#!/usr/bin/env bash
# 这个Shebang行是关键
# 使用env命令查找bash解释器，这比直接写#!/bin/bash更具可移植性
# 因为不同系统上bash的路径可能不同（如/bin/bash或/usr/bin/bash）

# 第二行开始是实际的脚本代码
echo "这个脚本将由bash解释执行"

# 注释行以#开头
# 注释可以占据整行，也可以在代码后面出现
some_command  # 这也是一个有效的注释
```

为什么要使用`#!/usr/bin/env bash`而不是直接写`#!/bin/bash`？原因在于可移植性[[[2](https://wxler.github.io/2021/02/10/111447/)]]。在某些系统上（如macOS或某些BSD系统），bash可能安装在不同的位置。使用`env`命令会自动在系统的PATH中搜索bash，这使得脚本在不同系统上都能正常工作。这是专业脚本编写中的最佳实践。

### 脚本执行的三种主要方式
在Linux中有三种主要的方式来执行Shell脚本，理解它们之间的区别对于正确使用脚本至关重要[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[14](https://blog.csdn.net/2301_80613136/article/details/140449795)]]。

第一种方式是作为可执行程序直接运行。这种方式需要脚本文件具有可执行权限，通过`chmod +x script.sh`来设置，然后使用`./script.sh`或完整路径如`/path/to/script.sh`来执行[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[14](https://blog.csdn.net/2301_80613136/article/details/140449795)]]。需要注意的是，不能直接写`script.sh`而必须写`./script.sh`，因为当前目录通常不在PATH环境变量中[[[4](http://www.runoob.com/linux/linux-shell.html)]]。

第二种方式是将解释器作为命令的参数。使用`bash script.sh`、`sh script.sh`或`/bin/bash script.sh`来执行脚本[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[14](https://blog.csdn.net/2301_80613136/article/details/140449795)]]。这种方式不需要脚本具有可执行权限，也不会读取脚本的第一行（Shebang行），而是直接使用指定的解释器[[[2](https://wxler.github.io/2021/02/10/111447/)]]。

第三种方式是使用`source`命令或`.`命令来执行脚本[[[2](https://wxler.github.io/2021/02/10/111447/)][[4](http://www.runoob.com/linux/linux-shell.html)][[14](https://blog.csdn.net/2301_80613136/article/details/140449795)]]。例如`source script.sh`或`. script.sh`。这种方式会在当前shell进程中执行脚本，而不是创建新的子shell。这意味着脚本中定义的变量和函数在脚本执行完毕后仍然存在于当前shell中，这在加载配置文件或函数库时非常有用[[[2](https://wxler.github.io/2021/02/10/111447/)]]。

```bash
#!/usr/bin/env bash
# 演示脚本执行方式的区别

# 这些变量会在不同的执行方式下有不同的作用域

export_var="这是一个导出变量"
local_var="这是一个本地变量"

# 如果这个脚本被直接执行或通过bash运行，会创建子shell
# 子shell中的变量不会影响父shell

# 如果这个脚本被source执行，则在当前shell中运行
# 所有变量都会保留在当前shell中
```

## 第二章 变量系统与参数扩展
### 变量的定义与使用基础
在Shell中定义变量的语法非常简单，但却容易出错[[[5](https://blog.csdn.net/m0_73345433/article/details/136459603)][[6](https://javaguide.cn/cs-basics/operating-system/shell-intro.html)]]。变量定义的基本格式是`variable_name=value`，其中等号两边都不能有空格[[[5](https://blog.csdn.net/m0_73345433/article/details/136459603)]]。这一点初学者经常犯错，导致脚本无法正常执行。

```bash
#!/usr/bin/env bash

# ✅ 正确的变量定义
name="张三"
age=28
path="/home/user/documents"

# ❌ 错误的定义方式（会导致命令未找到的错误）
# name = "张三"      # 等号两边有空格
# age= 28            # 等号后面有空格
# path =             # 等号前面有空格

# 访问变量的值使用$前缀
echo "我的名字是 $name"
echo "我的年龄是 $age"

# 推荐做法：使用${}来明确变量的作用范围
echo "我的名字是 ${name}"

# 这在变量名后紧跟其他字符时特别重要
# 如果没有{}，Shell可能会混淆变量名的边界
filename="report"
# 这会尝试访问变量$filename_txt（不存在）
# echo "生成的文件: $filename_txt"

# 使用{}可以正确访问
echo "生成的文件: ${filename}_txt"  # 输出: 生成的文件: report_txt
```

Shell中有三类变量[[[6](https://javaguide.cn/cs-basics/operating-system/shell-intro.html)]]：自定义变量（仅在当前shell中有效）、环境变量（可以传递给子进程）和Shell变量（由Shell程序自身设置的特殊变量）[[[6](https://javaguide.cn/cs-basics/operating-system/shell-intro.html)]]。自定义变量只在定义它们的shell进程中有效，不会自动传递给子进程[[[6](https://javaguide.cn/cs-basics/operating-system/shell-intro.html)]]。如果需要让子进程也能访问这个变量，必须使用`export`命令将其声明为环境变量[[[6](https://javaguide.cn/cs-basics/operating-system/shell-intro.html)]]。

```bash
#!/usr/bin/env bash

# 定义一个自定义变量
my_variable="仅在当前shell中有效"

# 定义并导出一个环境变量
export MY_ENV_VAR="可以传递给子进程"

# 如果先定义后导出
config_value="配置值"
export config_value  # 现在它可以被子进程访问

# 使用env命令查看所有环境变量
# env | grep MY_ENV_VAR

# 常用的环境变量
echo "用户主目录: $HOME"
echo "当前用户: $USER"
echo "当前工作目录: $PWD"
echo "Shell类型: $SHELL"
echo "PATH: $PATH"
```

### 特殊参数变量与命令行参数处理
Shell提供了多个特殊的参数变量来处理脚本的输入和执行状态[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)][[29](https://blog.csdn.net/qq_55908180/article/details/130751074)]]。这些变量在编写灵活的脚本时至关重要。

`$0`代表脚本本身的名称或被调用的命令名[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)]]。在直接执行时是完整路径或相对路径，使用`bash script.sh`执行时只是脚本名。`$1`、`$2`等代表传递给脚本的第一个、第二个参数，这些被称为位置参数[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)]]。`$#`代表传递给脚本的参数总数[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)]]。

```bash
#!/usr/bin/env bash

# 演示特殊参数变量

echo "脚本名: $0"
echo "参数个数: $#"
echo "第一个参数: $1"
echo "第二个参数: $2"

# 当参数超过9个时，必须使用大括号
echo "第十个参数: ${10}"
echo "第十一个参数: ${11}"

# $@和$*都代表所有参数，但在引用时有重要区别
echo "所有参数（$@）: $@"
echo "所有参数（$*）: $*"

# 这两者在被双引号包围时有不同的行为
# "$@"会将每个参数作为单独的字符串
# "$*"会将所有参数作为一个单一字符串

# 通过这个生产级案例来理解区别：
process_arguments() {
    echo "使用 $@ 进行遍历："
    for arg in "$@"; do  # 每个参数是独立的
        echo "  处理参数: $arg"
    done
    
    echo "使用 $* 进行遍历："
    for arg in $*; do    # 所有参数作为一个字符串
        echo "  处理参数: $arg"
    done
}

# 如果执行: ./script.sh "第一个" "第二个" "第三个"
# 使用"$@"会输出三行（分别是"第一个"、"第二个"、"第三个"）
# 使用$*会输出三行（如果参数中有空格，会被分解）
```

`$$`代表当前shell进程的进程ID（PID），这在需要创建唯一临时文件或标识进程时很有用[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)]]。`$?`代表上一个命令的退出状态码，0表示成功，非0表示失败[[[19](https://www.cnblogs.com/yikoulinux/p/15387440.html)][[22](https://blog.csdn.net/l_liangkk/article/details/105649018)]]。这是编写错误处理逻辑的关键。

```bash
#!/usr/bin/env bash

# 使用$$创建唯一的临时文件
temp_file="/tmp/backup_$$.tmp"

# 执行一个命令并检查其返回状态
if grep "error" /var/log/syslog > /dev/null 2>&1; then
    echo "找到错误"
else
    echo "命令退出状态: $?"  # 这会显示1（grep没有找到）
fi

# 完整的参数处理示例
echo "脚本被调用为: $0"
echo "当前进程ID: $$"
echo "参数个数: $#"

# 列出所有参数
if [[ $# -gt 0 ]]; then
    echo "接收到的参数:"
    for ((i=1; i<=\$#; i++)); do
        # 使用间接引用访问参数
        echo "  参数 $i: ${!i}"
    done
fi
```

### 参数扩展的高级用法
参数扩展是Shell语言中最强大的特性之一，允许对变量值进行复杂的操作，而无需调用外部命令[[[15](https://wangdoc.com/bash/expansion)][[18](https://blog.csdn.net/weixin_52575657/article/details/133753247)]]。这可以显著提高脚本的执行效率。

最基本的参数扩展是提供默认值。`${variable:-default}`的意思是如果变量未定义或为空，使用提供的默认值[[[18](https://blog.csdn.net/weixin_52575657/article/details/133753247)]]。`${variable:=default}`类似，但会同时设置变量的值[[[18](https://blog.csdn.net/weixin_52575657/article/details/133753247)]]。这两种方式都很常见，但它们的区别很重要。

```bash
#!/usr/bin/env bash

# 演示参数扩展的各种用法

# 1. 提供默认值
port="${PORT:-8080}"              # 如果PORT未定义，使用8080
timeout="${TIMEOUT:=30}"           # 同时设置TIMEOUT变量为30
echo "端口: $port, 超时时间: $timeout"

# 2. 检查变量是否存在
required_var="${REQUIRED:?错误：REQUIRED变量必须被定义}"
# 如果REQUIRED未定义，脚本会输出错误消息并退出

# 3. 只在变量存在且非空时使用值
result="${OPTIONAL:+存在的值}"     # 如果OPTIONAL存在且非空，使用"存在的值"

# 4. 获取变量长度
string="Hello World"
length="${#string}"
echo "字符串长度: $length"

# 5. 字符串截取
filename="archive_20240115_backup.tar.gz"
echo "完整文件名: $filename"
echo "文件名长度: ${#filename}"
echo "前5个字符: ${filename:0:5}"      # 从位置0开始，取5个字符
echo "从位置12开始: ${filename:12}"    # 从位置12到末尾
echo "倒数5个字符: ${filename: -5}"    # 注意这里有空格！

# 6. 删除字符串的前缀或后缀
path="/home/user/documents/file.txt"
# 删除最短匹配的前缀
directory="${path%/*}"              # 输出: /home/user/documents
# 删除最长匹配的前缀
just_filename="${path##*/}"         # 输出: file.txt

# 7. 删除文件扩展名和其他后缀
name_without_ext="${filename%.*}"   # 删除最短匹配的后缀（.gz）
name_with_date="${name_without_ext%_*}"  # 删除_及之后的内容

# 8. 字符串替换
text="数据库主机是localhost，应用主机也是localhost"
# 替换第一个匹配
replaced_once="${text/localhost/192.168.1.1}"
# 替换所有匹配
replaced_all="${text//localhost/192.168.1.1}"

echo "原文本: $text"
echo "替换一次: $replaced_once"
echo "替换全部: $replaced_all"

# 9. 生产级应用：配置文件路径处理
config_path="${CONFIG_DIR:-.}/app.conf"  # 默认使用当前目录
log_dir="${LOG_DIR:-.}/logs"
pid_file="${PID_DIR:-.}/${app_name}.pid"

echo "配置路径: $config_path"
echo "日志目录: $log_dir"
echo "PID文件: $pid_file"
```

参数扩展中最容易出错的地方是在使用负数索引时[[[10](https://www.ghosind.com/2020/10/12/shell-string)]]。`${string: -5}`表示从末尾向前数5个字符，但必须注意冒号和负号之间有一个空格。如果写成`${string:-5}`则会被解释为"如果变量未定义则使用-5"[[[10](https://www.ghosind.com/2020/10/12/shell-string)]]。

## 第三章 条件判断与测试表达式
### 测试命令的三种语法形式
Shell提供了三种测试条件的方式：`test`命令、方括号`[ ]`和双方括号`[[ ]]`[[[7](https://blog.csdn.net/m0_50668851/article/details/115122071)][[40](https://www.cnblogs.com/kaerxifa/p/12159562.html)][[43](https://blog.csdn.net/qq_47030101/article/details/143044979)]]。理解它们之间的区别和各自的适用场景至关重要。

```bash
#!/usr/bin/env bash

# 三种条件测试语法

# 方式1: test命令
if test -f /etc/hostname; then
    echo "hostname文件存在"
fi

# 方式2: 单方括号[ ]（POSIX兼容）
if [ -f /etc/hostname ]; then
    echo "hostname文件存在"
fi

# 方式3: 双方括号[[ ]]（仅Bash）
if [[ -f /etc/hostname ]]; then
    echo "hostname文件存在"
fi

# 关键区别演示

# 使用单方括号时容易出错
filename="my file.txt"

# ❌ 错误：不加引号会导致word splitting
# [ -f $filename ]  # 会被分解为两个文件

# ✅ 正确：必须加引号
[ -f "$filename" ]

# 使用双方括号则更安全
# ✅ 双方括号中可以不加引号（但为了代码一致性仍推荐加上）
[[ -f $filename ]]
[[ -f "$filename" ]]  # 两种都可以

# 双方括号支持正则表达式
if [[ $filename =~ \.txt$ ]]; then
    echo "这是一个文本文件"
fi

# 双方括号支持通配符
if [[ $filename == *.txt ]]; then
    echo "这是一个文本文件"
fi
```

双方括号`[[ ]]`是现代Bash编程中的首选[[[7](https://blog.csdn.net/m0_50668851/article/details/115122071)]]，因为它更安全、功能更强大。它支持正则表达式匹配、通配符、逻辑操作符`&&`和`||`，且内部的变量引用不需要加引号[[[7](https://blog.csdn.net/m0_50668851/article/details/115122071)]]。相比之下，单方括号`[ ]`会进行word splitting和filename expansion，这会导致许多隐藏的bug[[[7](https://blog.csdn.net/m0_50668851/article/details/115122071)]]。

### 文件测试运算符
Shell提供了多个文件测试运算符来检查文件的属性[[[9](https://cloud.tencent.com/developer/article/1838195)][[12](https://blog.csdn.net/m0_59091453/article/details/136983894)][[43](https://blog.csdn.net/qq_47030101/article/details/143044979)]]。这些测试在脚本中非常常用，特别是在处理文件和目录时。

```bash
#!/usr/bin/env bash

# 文件测试运算符详解

test_file="/etc/passwd"
test_dir="/home"
test_link="/bin/sh"  # 通常是一个符号链接
non_existent="/path/that/does/not/exist"

# 基本存在性测试
echo "=== 文件存在性测试 ==="

[[ -e "$test_file" ]] && echo "$test_file 存在"
[[ ! -e "$non_existent" ]] && echo "$non_existent 不存在"

# 文件类型测试
echo -e "\n=== 文件类型测试 ==="

[[ -f "$test_file" ]] && echo "$test_file 是普通文件"
[[ -d "$test_dir" ]] && echo "$test_dir 是目录"
[[ -L "$test_link" ]] && echo "$test_link 是符号链接"

# 权限测试
echo -e "\n=== 权限测试 ==="

[[ -r "$test_file" ]] && echo "$test_file 可读"
[[ -w "/tmp" ]] && echo "/tmp 可写"
[[ -x "/bin/bash" ]] && echo "/bin/bash 可执行"

# 文件大小测试
echo -e "\n=== 文件大小测试 ==="

[[ -s "$test_file" ]] && echo "$test_file 文件非空"
[[ ! -s "$test_file" ]] && echo "$test_file 文件为空"

# 获取文件大小的方法（虽然不是直接的文件测试）
file_size=$(stat -f%z "$test_file" 2>/dev/null || stat -c%s "$test_file" 2>/dev/null)
echo "$test_file 的大小: $file_size 字节"

# 文件的修改时间比较
file1="/etc/passwd"
file2="/etc/shadow"

if [[ "$file1" -nt "$file2" ]]; then
    echo "$file1 比 $file2 更新"
elif [[ "$file1" -ot "$file2" ]]; then
    echo "$file1 比 $file2 更旧"
else
    echo "$file1 和 $file2 修改时间相同"
fi

# 实用的文件检查函数
```

```bash

# 带步长的序列（仅在Bash 4+）

    echo "奇数: $i"
done

echo -e "\n=== 循环遍历带前缀的序列 ==="

# 生成带前缀的序列
for day in {Mon..Fri}; do
    echo "工作日: $day"
done

echo -e "\n=== 使用seq命令 ==="

# seq命令生成数字序列（当大括号扩展不可用时）
for i in $(seq 1 5); do
    echo "使用seq: $i"
done

# seq的高级用法：指定分隔符和步长
for i in $(seq 0 5 20); do
    echo "步长为5的序列: $i"
done

echo -e "\n=== C风格的for循环 ==="

# C风格的for循环语法
for ((i = 0; i < 5; i++)); do
    echo "C风格循环: $i"
done

# 复杂的C风格循环
for ((i = 10; i > 0; i--)); do

done

for ((i = 1, j = 10; i <= 5; i++, j--)); do
    echo "i=$i, j=$j"
done

done

for i in {1..3}; do
    for j in {1..3}; do
        echo "[$i, $j]"
    done
done

echo -e "\n=== 生产级案例：批量处理日志文件 ==="

process_log_files() {
    local log_dir="${1:-.}"
    local file_count=0
    local error_count=0
    
    # 使用for循环遍历所有日志文件
    for log_file in "$log_dir"/*.log; do
        # 检查是否真的存在文件（防止*.log未匹配时）
        [[ -f "$log_file" ]] || continue
        
        ((file_count++))
        echo "处理: $log_file"
        
        # 检查文件是否可读
        if [[ ! -r "$log_file" ]]; then
            echo "  警告: 文件不可读"
            continue

        # 计算文件中的错误行数
        error_lines=$(grep -c "ERROR" "$log_file" 2>/dev/null || echo "0")
        
        if (( error_lines > 0 )); then
            echo "  发现 $error_lines 行错误"
            ((error_count++))
        fi
    done
    
    echo "处理完成: 共处理 $file_count 个文件，$error_count 个文件包含错误"
}
```

```bash
retry_with_exponential_backoff() {
    local command="$1"
    local max_retries="${2:-5}"
    local initial_delay="${3:-1}"
    
    local retry_count=0
    local delay=$initial_delay
    
    while (( retry_count < max_retries )); do
        echo "[$(date '+%Y-%m-%d %H:%M:%S')] 尝试执行命令（第$((retry_count+1))/$max_retries次）"
        
        # 执行命令
        if eval "$command"; then
            echo "命令执行成功"
            return 0
        fi

            delay=$((delay * 2))
            if (( delay > 60 )); then
                delay=60  # 限制最大延迟
            fi
        fi
        
        ((retry_count++))
    done
    
    echo "达到最大重试次数，命令执行失败"
    return 1
}
```

```bash
    # 实现功能
    stat -f%z "$file" 2>/dev/null || stat -c%s "$file" 2>/dev/null
}

# 3. 使用local声明局部变量
calculate_factorial() {
    local n=$1
    local result=1
    
    if (( n < 0 )); then
        echo "错误：n必须是非负整数"
        return 1
    fi
    
    for ((i = 2; i <= n; i++)); do
        ((result *= i))
    done
    
    echo "$result"
}

# 4. 提供清晰的错误消息和返回码
safe_divide() {
    local dividend=$1
    local divisor=$2
    
    if [[ $# -ne 2 ]]; then
        echo "错误：需要两个参数" >&2
        return 1
    fi
    
    if ! [[ "$dividend" =~ ^[0-9]+$ ]] || ! [[ "$divisor" =~ ^[0-9]+$ ]]; then
        echo "错误：参数必须是整数" >&2
        return 2
    fi
    
    if (( divisor == 0 )); then
        echo "错误：除数不能为零" >&2
        return 3
    fi
    
    echo $((dividend / divisor))
    return 0
}

# 测试safe_divide
echo "10 / 2 = $(safe_divide 10 2)"
echo "10 / 0 的结果:"
safe_divide 10 0
echo "返回码: $?"

echo -e "\n=== 生产级案例：日志记录函数 ==="

    # 输出格式：[时间] [级别] 消息
    echo "[$timestamp] [$level] $message"
}

}
```

```bash
load_config() {

        echo "错误：配置文件不存在: $config_file"
        return 1
```

## 第七章 I/O重定向与管道
### 标准流的重定向
在Shell中，每个进程都有三个标准流：标准输入（stdin，文件描述符0）、标准输出（stdout，文件描述符1）和标准错误（stderr，文件描述符2）[[[14](https://blog.csdn.net/2301_80613136/article/details/140449795)][[17](https://www.cnblogs.com/hongyanee/p/3445068.html)]]。理解这些流的重定向是高效脚本编写的基础。

```bash
#!/usr/bin/env bash

echo "=== 输出重定向 ==="

# 将标准输出重定向到文件（覆盖）
echo "这会被写入文件" > /tmp/output.txt

# 将标准输出追加到文件
echo "这会被追加到文件" >> /tmp/output.txt

# 查看文件内容
echo "文件内容:"
cat /tmp/output.txt

echo -e "\n=== 错误输出重定向 ==="

# 将标准错误重定向到文件
cat /nonexistent/file.txt 2> /tmp/error.txt 2>/dev/null

# 将标准错误追加到文件
ls /nonexistent 2>> /tmp/error.log

# 如果错误文件存在，显示其内容
if [[ -f /tmp/error.txt ]]; then
    echo "错误输出内容:"
    cat /tmp/error.txt
fi

echo -e "\n=== 同时重定向标准输出和标准错误 ==="

# 方法1：分别重定向
command > /tmp/output.txt 2> /tmp/error.txt

# 方法2：将错误重定向到输出
command > /tmp/result.txt 2>&1

# 方法3：将两者都重定向到同一文件
command &> /tmp/combined.txt

# 方法4：将输出重定向到错误（不常用）
echo "这会被当作错误输出" 1>&2

echo -e "\n=== 输入重定向 ==="

# 从文件读取作为输入
# command < /tmp/input.txt

# Here Document：将多行文本作为输入
cat << EOF
这是第一行
这是第二行
这是第三行
EOF

# Here String：将字符串作为输入
wc -w <<< "这是一个字符串"

echo -e "\n=== 丢弃输出 ==="

# 使用/dev/null来丢弃不需要的输出
command > /dev/null 2>&1  # 丢弃所有输出

# 只查看错误，丢弃正常输出
command 2>&1 > /dev/null

echo -e "\n=== 文件描述符操作 ==="

# 复制文件描述符：将stdout复制给一个新的FD
exec 3>&1

# 使用新的FD输出
echo "这会输出到stdout（通过FD3）" >&3

# 关闭文件描述符
exec 3>&-

echo -e "\n=== 生产级案例：日志记录与错误处理 ==="

# 设置日志文件
LOG_FILE="/tmp/script_$$.log"
ERROR_LOG="/tmp/script_error_$$.log"

# 重定向所有输出
exec 1>> "$LOG_FILE"       # 将stdout重定向到日志
exec 2>> "$ERROR_LOG"      # 将stderr重定向到错误日志

echo "这是一条日志消息"
echo "这是一条错误消息" >&2

# 恢复标准输出用于显示消息
exec 1>&2  # 将stdout恢复到stderr（在脚本最后）

echo "脚本执行完毕"
echo "日志文件: $LOG_FILE"
echo "错误日志: $ERROR_LOG"
```

### 管道操作
管道是一个强大的特性，允许一个命令的输出作为另一个命令的输入[[[14](https://blog.csdn.net/2301_80613136/article/details/140449795)][[17](https://www.cnblogs.com/hongyanee/p/3445068.html)][[27](https://blog.csdn.net/lanse_rain/article/details/105969155)]]。

```bash
#!/usr/bin/env bash

echo "=== 基本管道操作 ==="

# 列出所有用户，显示用户名
cat /etc/passwd | cut -d: -f1

# 使用管道链
ps aux | grep "bash" | wc -l

echo -e "\n=== 管道中的命令执行 ==="

# 在管道中，每个命令在一个子shell中运行
count=0
echo -e "apple\norange\nbanana" | while read fruit; do
    ((count++))
    echo "处理: $fruit (计数: $count)"
done

# ❌ 注意：循环后count仍然为0（subshell中的修改不会保留）
echo "循环外的计数: $count"

echo -e "\n=== 避免subshell问题的方法 ==="

# 方法1：使用process substitution
count=0
while read fruit; do
    ((count++))
    echo "处理: $fruit (计数: $count)"
done < <(echo -e "apple\norange\nbanana")

echo "现在计数保留了: $count"

echo -e "\n=== tee命令：同时输出和保存 ==="

# tee命令将输入复制到stdout和指定的文件
echo -e "行1\n行2\n行3" | tee /tmp/tee_output.txt | wc -l

# 追加模式
echo "第四行" | tee -a /tmp/tee_output.txt

echo -e "\n=== 管道中的错误处理 ==="

# 注意：管道中只有最后一个命令的返回值被$?捕获
# 使用set -o pipefail可以改变这个行为

# 默认行为
nonexistent_command | cat
echo "返回值（默认）: $?"

# 使用pipefail
(set -o pipefail; nonexistent_command | cat)
echo "返回值（pipefail）: $?"

echo -e "\n=== 生产级案例：日志分析管道 ==="

# 创建示例日志文件
cat > /tmp/sample.log << 'EOF'
2024-01-15 10:20:30 INFO 用户登录成功
2024-01-15 10:20:45 ERROR 数据库连接失败
2024-01-15 10:21:00 INFO 请求处理完成
2024-01-15 10:21:15 WARN 磁盘空间不足
2024-01-15 10:21:30 ERROR 文件未找到
2024-01-15 10:21:45 INFO 备份开始
EOF

echo "日志分析结果："

# 统计每个级别的日志条数
echo "各级别日志统计:"
grep -o "ERROR\|WARN\|INFO" /tmp/sample.log | sort | uniq -c

# 提取所有错误

grep "ERROR" /tmp/sample.log | cut -d' ' -f4-

# 统计特定时间段的日志
echo -e "\n10:21时段的日志:"
grep "10:21" /tmp/sample.log | cut -d' ' -f3

# 复杂分析：找出最后的错误
echo -e "\n最后发生的错误:"
grep "ERROR" /tmp/sample.log | tail -1
```
