# 配置云服务器
## Linux 服务器 Swap 调优 
## 云服务器Linux系统设置虚拟内存（Swap）超详细教程
云服务器（主流Ubuntu/CentOS/Rocky Linux等Linux发行版）默认可能未开启Swap、或Swap分区过小，当物理内存不足时，设置合适的**虚拟内存（Swap）** 能避免进程被杀死、服务器卡顿崩溃。

### 核心前提
1. 云服务器推荐用**Swap文件**（而非Swap分区）：云服务器多为整盘弹性挂载，重新分区易破坏数据，Swap文件创建/调整/删除更灵活，性能与分区无差异；
2. Swap大小建议（按物理内存配比，云服务器无需过大，避免占用云盘IO）：
    1. 物理内存≤2G：配置**2G Swap**
    2. 2G＜物理内存≤8G：配置**4G Swap**
    3. 8G＜物理内存≤16G：配置**4-8G Swap**
    4. 物理内存＞16G：按需配置（4-8G即可，云服务器大内存场景一般无需Swap）
3. 适配系统：**Ubuntu 20.04/22.04/2024**（重点）、CentOS 7/8/9、Rocky Linux 8/9、Debian 11/12（所有命令稍作适配即可）；
4. 云盘要求：建议Swap文件建在**SSD云盘**上，HDD云盘IO性能低，用作Swap会拖慢服务器。

## 一、先检查当前Swap状态（必做）
执行以下命令，查看服务器是否已开启Swap、及现有Swap大小：

```plain
# 查看Swap详细信息（大小、类型、是否启用）
free -h
# 或更精准的Swap状态查询
swapon --show
```

### 结果解读
+ 若`free -h`中**Swap行total为0**、`swapon --show`无输出：说明未开启Swap（云服务器默认多为这种情况）；
+ 若Swap大小远小于上述建议值：需扩容Swap。

## 二、创建并启用Swap文件（通用步骤，Ubuntu优先）
全程用`sudo`提权（root用户可直接去掉`sudo`），**命令可直接复制执行**，以下以**创建4G Swap**为例（需调整大小可直接改数字）。

### 步骤1：创建Swap文件（快速创建，推荐fallocate）
`fallocate`是快速创建大文件的工具，比`dd`命令快，优先使用：

```plain
# 创建4G的Swap文件，路径/swapfile（固定路径，无需修改）
sudo fallocate -l 4G /swapfile
```

#### 若执行报错`fallocate: fallocate failed: Operation not supported`（部分云服务器文件系统不支持），改用dd命令：
```plain
# 同样创建4G Swap文件，耗时稍长，耐心等待即可
sudo dd if=/dev/zero of=/swapfile bs=1G count=4
```

+ 调整大小：如创建2G则改`4G`为`2G`、`count=4`为`count=2`；创建8G则改`8G`、`count=8`。

### 步骤2：设置Swap文件权限（关键，必须600）
Swap文件涉及系统内存管理，需限制仅root用户可读写，否则系统会报权限警告、无法启用：

```plain
# 设置权限为600（仅root读、写）
sudo chmod 600 /swapfile
# 验证权限（输出-rw-------即正确）
ls -l /swapfile
```

### 步骤3：格式化Swap文件（将文件标记为Swap格式）
让系统识别该文件为虚拟内存分区，执行专用的mkswap命令：

```plain
sudo mkswap /swapfile
```

#### 成功输出示例：
```plain
Setting up swapspace version 1, size = 4 GiB (4294967296 bytes)
no label, UUID=xxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

### 步骤4：立即启用Swap文件
格式化后，执行命令让系统立即加载并使用该Swap，无需重启服务器：

```plain
sudo swapon /swapfile
```

### 步骤5：验证Swap是否启用成功
重新执行状态查询命令，确认Swap已生效：

```plain
free -h
swapon --show
```

#### ✅ 成功标志：
+ `free -h`中**Swap行total显示为设置的大小**（如4G），used为0（刚启用暂无使用）；
+ `swapon --show`输出Swap文件路径`/swapfile`、类型`file`、大小正确。

## 三、设置Swap开机自动挂载（核心，避免重启失效）
云服务器重启后，临时启用的Swap会失效，需将Swap文件写入系统挂载配置文件`/etc/fstab`，让系统开机自动加载。

### 步骤1：编辑fstab配置文件
用nano编辑器（新手友好）打开文件，nano比vim简单，无需记复杂快捷键：

```plain
sudo nano /etc/fstab
```

### 步骤2：添加Swap挂载配置
在文件**最后一行**，粘贴以下内容（直接复制，无需修改），按**Ctrl+O**保存，**Ctrl+X**退出编辑器：

```plain
/swapfile none swap sw 0 0
```

### 配置说明
+ `/swapfile`：Swap文件路径，与创建的一致；
+ `none`：无挂载点，Swap无需挂载点；
+ `swap`：文件系统类型为swap；
+ `sw`：Swap专用挂载选项；
+ `0 0`：表示开机不备份、不检查，符合Swap规范。

### 步骤3：验证fstab配置（避免开机报错，必做）
执行挂载检测命令，检查配置是否有语法错误，**无任何输出即配置正确**：

```plain
sudo mount -a
```

⚠️ 若有报错：立即重新编辑`/etc/fstab`，检查是否多打/少打字符（如空格、标点）。

## 四、优化Swap性能：调整swappiness（云服务器必改）
`swappiness`是Linux内核参数，控制**系统使用Swap的积极程度**，取值0-100：

+ 数值越高：系统越容易将物理内存中的数据换到Swap中；
+ 数值越低：系统越优先使用物理内存，仅在物理内存不足时使用Swap。

### 云服务器优化建议
云服务器的Swap建在云盘上，IO性能远低于物理内存，**建议将swappiness设为10-20**：优先用物理内存，仅物理内存占用≥90%时才使用Swap，避免频繁的内存/云盘数据交换拖慢服务器。

### 步骤1：临时调整swappiness（立即生效，重启失效）
```plain
# 设为10，Ubuntu/CentOS通用
sudo sysctl vm.swappiness=10
```

#### 成功输出：`vm.swappiness = 10`
### 步骤2：永久调整swappiness（重启生效，核心）
将参数写入内核配置文件`/etc/sysctl.conf`，让开机后永久生效：

```plain
# 追加参数到文件最后一行
sudo echo "vm.swappiness=10" | sudo tee -a /etc/sysctl.conf
# 重新加载配置，让永久设置立即生效
sudo sysctl -p
```

### 验证：
```plain
cat /proc/sys/vm/swappiness
```

输出`10`即设置成功。

## 五、后续常用操作：扩容/删除Swap（备用）
### 场景1：扩容Swap（如从4G扩到8G）
```plain
# 1. 先关闭现有Swap
sudo swapoff /swapfile
# 2. 扩容文件（如扩到8G，fallocate直接改大小）
sudo fallocate -l 8G /swapfile
# 3. 重新格式化
sudo mkswap /swapfile
# 4. 重新启用
sudo swapon /swapfile
# 5. 验证
free -h
```

### 场景2：删除Swap（无需使用时）
```plain
# 1. 关闭Swap
sudo swapoff /swapfile
# 2. 删除Swap文件
sudo rm -f /swapfile
# 3. 编辑fstab，删除最后一行的Swap配置
sudo nano /etc/fstab
# 4. 保存退出后，验证（无Swap输出即成功）
free -h
```

## 六、CentOS/Rocky Linux适配说明
上述命令**99%适配CentOS 7/8/9、Rocky Linux 8/9**，仅需注意1个点：

+ 若CentOS最小化安装，可能缺少`fallocate`工具，直接用`dd`命令创建Swap文件即可（步骤1中的备用命令）；
+ 其余所有步骤（权限、格式化、启用、fstab配置、swappiness调整）完全一致。

## 七、云服务器设置Swap注意事项
1. 不要将Swap设得过大：云盘IO是瓶颈，过大的Swap不仅无用，还会因频繁IO导致服务器卡顿；
2. 优先升级物理内存：Swap仅为**应急方案**，若服务器频繁占用Swap，说明物理内存不足，云服务器直接升级内存是根本解决办法；
3. 避免在临时云盘创建Swap：部分云服务器有临时盘（如/tmp），重启后数据丢失，Swap文件必须建在**持久化云盘**（如/根目录）；
4. 定期检查Swap使用：若Swap长期高占用（≥50%），立即排查服务器进程（用`top/htop`命令），是否有内存泄漏的程序。

## 最终验证
执行以下命令，若所有输出均符合预期，说明Swap设置完全成功（开机自动生效+性能优化）：

```plain
# 1. Swap状态
free -h
# 2. Swap挂载
swapon --show
# 3. swappiness值
cat /proc/sys/vm/swappiness
# 4. fstab配置无错误
sudo mount -a
```

# docker配置 系统可以访问访问外网，但是无法走代理
之前执行`docker info | grep proxy`可能显示代理配置，但Docker**实际未加载**（系统d配置重载不彻底导致），需用更精准的命令验证，并重载重启Docker：

### 1.1 检查Docker服务的环境变量（真正的代理加载凭证）
```bash
# 查看Docker守护进程是否真的获取到代理环境变量
sudo systemctl show --property=Environment docker
```

✅ **成功标志**：输出包含`HTTP_PROXY=http://127.0.0.1:7890`和`HTTPS_PROXY=http://127.0.0.1:7890`，如下：

```plain
Environment=HTTP_PROXY=http://127.0.0.1:7890 HTTPS_PROXY=http://127.0.0.1:7890 NO_PROXY=127.0.0.1,localhost,...
```

❌ **若未显示/显示1080端口**：说明代理配置完全没加载，执行下一步重新配置。

### 1.2 重新配置Docker代理（覆盖原有配置，确保无格式错误）
```bash
# 先删除原有代理配置文件（避免格式冲突）
sudo rm -f /etc/systemd/system/docker.service.d/http-proxy.conf
# 重新创建并写入配置（无多余空格，格式绝对正确）
sudo tee /etc/systemd/system/docker.service.d/http-proxy.conf <<-'EOF'
[Service]
Environment="HTTP_PROXY=http://127.0.0.1:7890"
Environment="HTTPS_PROXY=http://127.0.0.1:7890"
Environment="NO_PROXY=127.0.0.1,localhost,*.cn,*.aliyuncs.com,*.qiniu.com,192.168.0.0/16,10.0.0.0/8"
EOF
```

### 1.3 强制重载+重启Docker（比普通reload更彻底，确保加载配置）
```bash
# 强制重载systemd（清空缓存，重新识别配置）
sudo systemctl daemon-reload
# 停止Docker并彻底重启（而非restart，避免残留进程）
sudo systemctl stop docker && sudo systemctl start docker
# 再次验证代理环境变量
sudo systemctl show --property=Environment docker
```

**必须确保此步骤输出包含7890端口的代理配置，再进行下一步**。

## 步骤2：验证mihomo是否正常代理，且对Docker域名做分流
Docker代理配置生效后，若仍超时，说明**mihomo本身未正常转发请求**，或**规则把docker.io导去了直连**，需验证mihomo的连通性和分流规则：

### 2.1 先验证mihomo终端代理是否正常（基础保障）
确保当前终端代理已配置7890端口（步骤1已做），测试访问Docker官方域名：

```bash
# 测试访问Docker官方仓库，返回401=网络通（未认证，正常）
curl -v https://registry-1.docker.io/v2/
# 测试mihomo是否真的走代理IP（返回非云服务器IP=成功）
curl ipinfo.io/ip
```

✅ 若`curl`能通（401），说明mihomo代理本身正常；若`curl`也超时，先重启mihomo：

```bash
sudo systemctl restart mihomo && sudo systemctl status mihomo
```

### 2.2 确保mihomo规则对docker.io做代理分流（关键）
你的mihomo配置用了`RULE-SET,main,DIRECT`规则集，默认会对**国外域名（含docker.io）走PROXY**，但需确认配置中`MATCH,PROXY`兜底规则存在（之前的配置模板已包含），无需修改配置，只需**手动更新一次规则集**：

```bash
# 手动更新mihomo规则集，确保分流规则最新
sudo mihomo -d /etc/mihomo -u && sudo systemctl restart mihomo
```

## 步骤3：兜底方案——配置Docker强制走终端代理（100%生效）
若以上两步仍未解决，用**Docker临时挂载代理**的方式，让`docker pull`强制走终端的7890代理，这是云服务器的兜底方案，无视Docker守护进程的代理配置：

### 3.1 确保当前终端代理已生效（7890端口）
```bash
# 重新配置终端代理（确保无残留）
unset http_proxy https_proxy all_proxy
export http_proxy=http://127.0.0.1:7890
export https_proxy=http://127.0.0.1:7890
export all_proxy=socks5://127.0.0.1:7890
```

### 3.2 用`sudo -E`保留终端代理环境执行docker pull（核心）
`sudo -E`会让`sudo`命令**继承当前终端的所有环境变量（含代理）**，强制Docker走终端的7890代理，这是最直接的生效方式：

```bash
# 用sudo -E执行拉取，保留终端代理（100%走mihomo）
sudo -E docker pull redis
```

✅ 此命令若能正常拉取（进度条走动），说明之前的Docker守护进程代理配置有问题，可长期用此方式执行Docker命令，或继续下一步修复。

## 步骤4：额外优化——配置Docker DNS，避免域名解析超时（云服务器必做）
云服务器可能存在**DNS污染**，导致解析`registry-1.docker.io`时返回错误IP，即便代理生效也会超时，需给Docker配置**国内纯净DNS**（如阿里云、腾讯云DNS）：

### 4.1 创建Docker DNS配置文件
```bash
# 创建Docker主配置文件（若不存在）
sudo mkdir -p /etc/docker
sudo nano /etc/docker/daemon.json
```

### 4.2 粘贴配置（合并镜像加速器+DNS，若已有内容则追加DNS）
```json
{
  # 若之前配了七牛云/阿里云加速器，保留即可，新增dns节点
  "registry-mirrors": ["https://reg-mirror.qiniu.com"],
  "dns": ["223.5.5.5", "119.29.29.29", "8.8.8.8"]
}
```

+ 223.5.5.5/119.29.29.29：阿里云/腾讯云国内DNS，解析快且防污染；
+ 8.8.8.8：Google DNS，兜底解析国外域名。

### 4.3 重启Docker让DNS生效
```bash
sudo systemctl daemon-reload && sudo systemctl restart docker
```

## 常见避坑总结（云服务器Docker+代理必看）
1. **Docker代理配置≠生效**：`docker info | grep proxy`仅显示配置文件存在，`sudo systemctl show --property=Environment docker` 才是真正验证是否加载的命令；
2. **sudo会清空终端代理**：普通`sudo docker pull`会丢失终端代理环境，必须用`sudo -E`保留；
3. **DNS污染是隐形坑**：即便代理生效，解析docker.io出错也会超时，必须配置Docker国内DNS；
4. **mihomo规则需兜底PROXY**：配置中必须有`MATCH,PROXY`，确保所有未匹配规则的域名都走代理，避免国外域名漏网。

---

## 长期使用建议
1. 若想**无需sudo**执行Docker且走代理，确保用户在docker组+终端代理生效，用`docker pull`即可；
2. 若想**彻底不用sudo**，且Docker守护进程代理生效，确保步骤1的Docker环境变量加载成功，后续可正常执行所有Docker命令；
3. 定期更新mihomo规则集：`sudo mihomo -d /etc/mihomo -u && sudo systemctl restart mihomo`，避免分流规则过时。

至此，你的Docker即可通过mihomo代理正常拉取国外官方镜像，结合之前的国内Docker源，实现「国内源安装Docker+代理拉取国外镜像」的双重保障。

columns = [
    {"fieldname": "table_info", "label": "表名 (中英对照)", "fieldtype": "Data", "width": 280},
    {"fieldname": "fieldname", "label": "数据库列名 (Fieldname)", "fieldtype": "Data", "width": 180},
    {"fieldname": "label_zh", "label": "字段中文名", "fieldtype": "Data", "width": 150},
    {"fieldname": "label_en", "label": "字段英文名", "fieldtype": "Data", "width": 150},
    {"fieldname": "fieldtype", "label": "字段类型 (Type)", "fieldtype": "Data", "width": 120},
    {"fieldname": "options", "label": "选项/关联表", "fieldtype": "Data", "width": 200},
    {"fieldname": "description", "label": "描述说明", "fieldtype": "Data", "width": 250}
]

result_data = []

# 从底层捞取所有纯英文字段
fields = frappe.db.get_all("DocField",
                           fields=["parent", "fieldname", "label", "fieldtype", "options", "description"],
                           filters={"fieldtype": ["not in", ["Section Break", "Column Break", "Tab Break", "HTML"]]},
                           order_by="parent asc, idx asc"
                          )

# 循环每一行进行翻译和组合
for f in fields:
    # 翻译中文
    translated_parent = _(f.parent) if f.parent else ""
    translated_label = _(f.label) if f.label else ""
    translated_desc = _(f.description) if f.description else ""

    # 组合表名格式，例如：销售订单明细（tabSales Order Item）
    # 如果该表没有对应的中文翻译，就直接显示 英文（tab英文）
    if translated_parent and translated_parent != f.parent:
        combined_table_name = f"{translated_parent}（tab{f.parent}）"
    else:
        combined_table_name = f"{f.parent}（tab{f.parent}）"

    result_data.append({
        "table_info": combined_table_name,
        "fieldname": f.fieldname,
        "label_zh": translated_label,
        "label_en": f.label,
        "fieldtype": f.fieldtype,
        "options": f.options,
        "description": translated_desc
    })

# 将列定义和数据打包返回给框架
data = columns, result_data

### 🎯 方法一：快速测试法 (仅推荐用于个人、临时环境)
这是最快的验证方法，但**仅适用于个人临时测试环境**，因为它会降低浏览器的安全级别。

1. **复制你的访问地址**：从浏览器地址栏复制 `code-server` 的完整访问地址，例如 `http://192.168.1.100:8080`。
2. **找到浏览器“实验”功能**：在地址栏输入并访问：
    - **Chrome/Edge**: `chrome://flags/#unsafely-treat-insecure-origin-as-secure`
    - **Firefox**: 在地址栏输入 `about:config`，搜索并启用 `security.insecure_field_warning.contextual.enabled`，但此方法效果有限，更推荐上述 Chrome/Edge 方案。
3. **添加安全例外**：将你的访问地址粘贴到输入框中，在下拉列表中选择 `**Enabled**`。
4. **重启浏览器**：点击页面底部的 **Relaunch** 按钮，浏览器会重启并生效。
